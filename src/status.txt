MEGA8086 Emulator — Investigation Status
==========================================
Last updated: 2026-04-01

FREEDOS "SECTOR" HANG — WHAT WE KNOW
======================================

1. FreeDOS boots through kernel init, prints "InitDisk",
   "illegal partition table - drive 00 sector" and then hangs.
   It never prints the sector number "0" or continues.

2. The hang is a loop in valid kernel code at CS=$0F75, IP in
   the $FC00-$FD00 range. The code cache loads real (non-zero)
   instructions from attic for those pages. The CPU is executing
   real code, not zeros.

3. The loop is in the FreeDOS printf engine. It processes the
   "%lu" format specifier for the sector number. The format
   dispatcher uses:
   - DS:[BX+$1095] to look up format character indices
   - CS:[BX+$F219] as a jump table to dispatch format handlers

4. At hang time, DS=$9DCA (not $0F75). DS=$9DCA equals SS.
   On the working emulator at the A:> prompt, DS=CS=$2902.

5. The CS-based jump table at CS:$F219 reads as all zeros through
   the emulated memory path. On the working emulator, CS:$F219
   has non-zero data.

6. DS=$0F75 (correct) at the moment CS first becomes $0F75.
   CS enters $0F75 exactly once and never leaves.
   No MOV sreg, POP DS, or LDS instruction changes DS while
   CS=$0F75 (logged and confirmed).

7. DS changes from $0F75 to $9DCA without any tracked 8086
   instruction causing the change. All tracked write paths to
   reg_ds (MOV sreg, POP seg, LDS, write_rm8, write_rm16)
   were monitored and none fired for DS while CS=$0F75.

8. The mark_cache_dirty function was confirmed to corrupt ZP
   via cache_dirty,x with out-of-range X during early boot
   (before CS=$0F75 was entered). Specifically, sta $98,x with
   X=$80 wraps in ZP and writes to $18 (reg_ds). This was caught
   by a targeted trap ($9B6A=$A2 after FLAGS push in
   do_sw_interrupt).

9. Adding bounds checks to mark_cache_dirty prevented the early
   corruption but did NOT fix the "sector" hang. DS still ends
   up as $9DCA at hang time, suggesting a second corruption
   source exists.

10. The corruption is not detected by watchpoints at:
    - Top of main loop (every iteration)
    - opcode_done (after every instruction)
    - REP loop iterations
    - do_sw_interrupt entry and after each push_word
    - write_rm8/write_rm16 register paths
    This suggests the corruption occurs within instruction
    execution or helper functions called during instruction
    execution, but after all current check points.

11. The working emulator (IBM PC AT 5170 with same FreeDOS disk)
    boots to A:\> successfully. It uses CS=$2902 (different from
    our CS=$0F75 due to different memory size/layout).

KERNEL BOOT SEQUENCE (confirmed)
=================================

- Boot sector loads KERNEL.SYS to ES=$1FE0 (linear $1FE00)
- UPX decompressor runs at CS=$0060, decompresses kernel in-place
  (~57KB of decompressed data in bank 4 at $0600-$FFFF)
- Kernel Phase 1 relocation runs at CS=$0256:
  - First REP MOVSW (backward, DF=1): copies INIT segment to
    ES=$9A04 (near top of 640KB)
  - Second REP MOVSW (backward, DF=1): copies HMA_TEXT to
    ES=$9001
  - Both complete successfully (confirmed by completion counter)
- Total of 22 REP MOVSW and 170+ REP MOVSB completions during boot
- Later MoveKernel() copies relocate the resident kernel to
  CS=$0F75 (base $F750)

WHAT IS NOT THE CAUSE
======================

- Cache flush mechanism (verified working via DEBUG test)
- Cache capacity (4 and 8 lines produce same result)
- INT 8 timer delivery (suppressing it doesn't fix the hang)
- Missing 8086 opcodes (unimpl_count=2, both FPU ESC)
- Divide overflow (counter stays at 0)
- Port polling ($3DA/$3BA read counter = 0)
- Code fetch from wrong bank (full-linear fetch fix applied)
- INT 13h sector copy boundary crossing (safe copy path applied)

REMAINING CONFIRMED BUGS
==========================

- mark_cache_dirty can write to arbitrary ZP via cache_dirty,x
  when X is out of range (0-3 expected). This is because
  temp_ptr+1 can be outside the CACHE_BUF range ($92-$95),
  causing the subtraction temp_ptr+1 - $92 to produce a
  large X value that wraps in ZP indexed addressing.
  This bug was confirmed to corrupt reg_ds during early boot.
  A second instance of this or similar corruption likely exists
  but has not been identified.

- CS jump table at CS:$F219 contains zeros on our emulator but
  real data on the working emulator. Whether this is caused by
  the DS corruption (wrong data reads) or by missing data
  population has not been determined.
