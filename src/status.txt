===========================
Last updated: 2026-04-26

RECENT CHANGES (2026-04-24 .. 2026-04-26 — ChatGPT-driven bug-find sweep)
==========================================================================

Long collaborative session focused on hardening MegaDOS (COMMAND.COM)
against subtle correctness bugs flagged by ChatGPT review. Each fix
was paired with a FIXTEST regression where practical; tests now go
from T1..T15. A few emulator-side issues were also caught.

COMMAND.COM / MegaDOS DOS handler fixes
----------------------------------------

SET / PROMPT / PATH atomic env updates (af37075)
  Old code deleted the existing variable before proving the new
  value fit, so an oversized new value left the user with neither.
  Now snapshot the env block at command entry and restore it from
  the overflow path. New env_backup_buf (ENV_SIZE) + env_backup /
  env_restore helpers.

AH=3Dh open: mask sharing/inherit bits (e7fc3e2)
  AL is the full DOS open-mode byte (access bits 0-2, sharing 4-6,
  inherit bit 7). Storing the raw byte made e.g. 0x80|read land as
  0x81, which then sailed past AH=40h's exact `cmp byte, 1`
  read-only guard. Validate low nibble (access + reserved bit 3)
  before masking. Also fixed a long-standing pop-imbalance in
  .i21_3d_no_handle (popped 6 vs 5 pushes — would have eaten the
  IRET frame's IP if FILES= ever exhausted).
  Test: T8.

AH=3Ch/3Dh/41h refuse directories, VOL labels, RO files (c1a68b1)
  AH=3Ch silently truncated whatever the name pointed at; AH=3Dh
  opened directories/volume label as files and ignored RO bit on
  write/rw opens; AH=41h would unlink directory entries (and free
  the cluster chain). Each handler now checks the attribute byte
  and rejects with AX=5. AH=3Dh still permits RO files for read.
  Test: T9.

AH=42h seek: don't trample last cluster on seek-to-EOF (afd710d)
  When the FAT-chain skip walk hit EOF before finishing, the
  handle was left at "current = last real cluster, pos_in_cluster
  = 0". A subsequent AH=40h append on a file whose size was a whole
  multiple of the cluster size then read that last cluster back
  and overwrote its first bytes. Fix: leave current at last real,
  bump pos_in_cluster to bpc — the existing AH=40h flush_next /
  allocate-new-cluster path takes over correctly.
  Test: T10.

AH=40h CX=0 truncate frees the chain (6a7580a)
  Old code only updated the SFT size; FAT clusters past the new
  EOF stayed allocated (leak), and truncate-to-zero left
  start_cluster pointing at an orphaned chain. New path computes
  ceil(P/bpc) clusters to keep, marks the last as EOF, frees the
  successor chain, and on truncate-to-zero clears start/current
  cluster. For partial truncate, current_cluster is repinned to
  the last kept cluster + pos_in_cluster recomputed (using bpc on
  a boundary, matching AH=42h's past-EOF convention) so a follow-up
  write doesn't trample a freed cluster.
  Same commit also fixed AH=36h (get free space) clobbering ES
  with SHELL_SEG and not restoring it (broke any post-AH=36h
  rep stosb in the caller's segment), and capped the redir_*_fname
  78-byte buffers in the parser so a long redirection target can't
  overflow.
  Test: T11 (three parts: truncate-to-zero, partial truncate, and
  truncate-then-append-on-same-handle).

AH=56h rename, three rounds of fixes:
  • Reject duplicate destinations (7c2d64f) — old code searched
    for the source 8.3 and overwrote it in place without ever
    checking whether the new name already existed. Two entries
    with the same 8.3 = ambiguous lookups. Same commit also
    cleaned up a latent ES bug: handler used ES:DI for internal
    rep movsb / parse_83_filename rep stosb but ES was the
    caller's segment. Worked from the shell prompt (ES = SHELL_SEG)
    but silently scribbled into the caller's PSP from a .COM. Pin
    ES = SHELL_SEG once at handler entry; remove the stray
    `mov es, [.i21_56_new_seg]` that was undoing the fix.
  • Reject path-qualified destinations (05631e7) — parse_83_filename
    bails on the first '\\', so "\\DIR\\NEW.TMP" or "A:NEW.TMP" left
    exec_fname all-spaces and the rename overwrote the source
    entry's name with blanks. Scan for '\\' or ':' before parsing;
    return AX=3 on hit.
  • Search source before same-name / dup checks (f5c12f8) — pre-fix
    MISSING -> MISSING returned success (same-name short-circuit
    fired without verifying the source existed) and MISSING ->
    EXISTING returned AX=5 instead of AX=2. Reorder: source-search
    first, stash dir-entry pointer, then same-name + dup checks.
  Test: T12 (six parts — A: dup reject, B: same-name no-op, C:
  path-qualified reject, D: missing→missing, E: missing→existing,
  F: positive-path success).

AH=3C/3D skip handles whose JFT slot is a live DUP alias (1b82f01)
  Architecture: handle == SFT index. AH=45h DUP creates a JFT
  alias (e.g. JFT[6] = 5) without allocating SFT slot 6, leaving
  it empty. AH=3C/3D's free-handle scan checked only file_handles
  (the SFT slot) so the next open could claim handle 6, overwrite
  JFT[6], and strand refcount on SFT 5 — stranded SFT, dirty data
  never finalized. Now require BOTH the SFT slot AND JFT[bx] to be
  free.
  Test: T13 (two parts — AH=3D path and AH=3C path).

AH=3C/39 refuse empty / path-only inputs (f8dd9bc)
  resolve_path returns CF=0 with exec_fname all-spaces when input
  is empty or ends in '\\'. Both create paths went on to write a
  blank 8.3 directory entry. Add a `cmp byte [exec_fname], ' '`
  guard after resolve_path. AH=5B inherits the fix (delegates to
  AH=3C). Other resolve_path callers (CHDIR, FindFirst, etc.) keep
  the old behavior because they need the directory-only resolution.
  Test: T14.

term_common: close child JFT[0..MAX] + inherit symmetry (3346236)
  Children could DUP2 a file SFT onto stdin/stdout/stderr and exit
  without closing — term_common only walked handles 5..MAX so the
  alias leaked, sft_finalize never ran, dir entry stayed at size 0.
  Fixed term to close 0..MAX. The matching change had to be made
  in BOTH inherit-refcount paths (the shell-launched setup AND the
  AH=4Bh-specific setup) to bump refcount[SFT 0..4] symmetrically;
  otherwise the new term decrements would underflow std-handle
  refcounts past the parent's working level and stdout would
  produce garbage on return. (Diagnosed live during T15
  development — required matching exec_inherit on both sides.)
  Test: T15 + T15CHLD.COM (a 50-byte child that DUP2s file→stdout
  and exits without close; T15 EXECs it and asserts the file dir
  entry shows size 4, not 0). T15 also illustrates two non-obvious
  test-side requirements: (a) a .COM exec'ing another .COM must
  call AH=4Ah to shrink its MCB first since shell-launched .COMs
  own all available memory, and (b) AH=4B leaves DS=SHELL_SEG on
  both success and failure, so callers must reset DS *before*
  checking CF if they print error messages.

AH=45/46 honor runtime files_limit (6f81e66)
  Both DUP loops scanned/validated against MAX_HANDLES instead of
  [cs:files_limit], so a CONFIG.SYS FILES= setting below
  MAX_HANDLES could yield handles other I/O calls reject as
  invalid. One-line fixes in each. (No test — the bug is only
  observable when FILES= < MAX_HANDLES, and stock CONFIG.SYS has
  FILES=20 == MAX_HANDLES.)

Emulator-side fixes
-------------------

io.asm: CRTC index/data port reads (3f7690c)
  Reads of $3D4/$3D5 and $3B4/$3B5 fell through to _irp_cga_other
  and unconditionally returned 0. Guests doing OUT 3D4,X / IN AL,
  3D5h to read back cursor or save/restore CRTC state saw zeros.
  Now $3D4/$3B4 reads return the io_crtc_reg latch; $3D5/$3B5
  reads return io_save_a/io_save_b for registers 0Eh/0Fh, 0 for
  others.
  Also stopped overloading io_crtc_reg as a screen-code scratch
  in INT 10h AH=09/0A — added io_int10_char so an INT 10h call
  between a guest's OUT 3D4 and OUT/IN 3D5 can't clobber the
  index latch.

save_floppy_drive: load filename from per-drive slot (9d172df)
  floppy_fname_page is shared with menu_read_filename, so the
  most recent mount/prompt could leave a stale name there. Saving
  drive A after mounting B would write A's contents to B's image
  file. Repopulate from drive_a_fname / drive_b_fname based on
  the drive arg before calling Hyppo setname; refuse outright if
  the per-drive slot is empty (Hyppo setname on zero-length
  silently no-ops, which would have made the save operate on
  whatever filename was previously set).

menu_tab_handler: flush + invalidate data cache before menu (0202ac5)
  Cache backing store ($9200-$95FF) and the cache tags / dirty
  bits in ZP $90-$9C are both fair game for KERNAL while the menu
  is up. The TAB handler saved ZP without flushing or invalidating,
  so on resume the saved cache tags still claimed lines were dirty
  and the next eviction would DMA whatever bytes the menu/KERNAL
  left in $9200 back to attic — silently corrupting guest RAM.
  Fix: cache_flush_all then cache_invalidate_all before saving ZP.
  Cost is paid once per menu entry, invisible to normal emulation.

Performance / cleanup
---------------------

cursor_update: drop the 32-bit multiplier for scr_col*8 (bded72d)
  scr_col tops out at 79 so scr_col*8 fits in 10 bits — three
  asl/rol pairs replace 8 multiplier setup writes + 2 result
  reads. Roughly cycle-neutral but frees the multiplier and
  reads cleaner.

Remove dead sti_shadow_flag store
  decode.asm:53 saved raw_opcode into sti_shadow_flag every
  emulated instruction. grep shows zero readers — the actual
  STI/MOV-SS/POP-SS shadow runs through irq_inhibit. Removing
  the lda+sta saves 8 cycles per emulated 8086 instruction in
  the hottest path in the emulator. The variable definition in
  globals.asm was also cleaned up so $8F00 is freed.

Drop per-instruction update_opcode_ptr from ml_next (af19f05)
  ml_next was unconditionally calling update_opcode_ptr on
  every non-attic emulated instruction (~70 cycles for 8-bit
  adds + region classification + jsr/rts overhead). fetch_byte
  already keeps opcode_ptr in lockstep with reg_ip via inline
  increment + bank-boundary resync + IP-wrap resync, and every
  IP/CS-modifying opcode handler explicitly calls
  update_opcode_ptr (or compute_cs_base + update_opcode_ptr)
  before returning. The per-instruction recompute was pure
  waste in the hottest path in the emulator.

  Audit covered every reg_ip / reg_cs writer in opcodes.asm
  and decode.asm: Jcc, JMP near/short/far, CALL near/far,
  indirect JMP/CALL via FF group, RET/RETF, IRET, LOOP /
  LOOPZ / LOOPNZ / JCXZ, REP retry rewind, INT* (via
  do_sw_interrupt which itself does both calls). All have
  explicit update_opcode_ptr.

  Three changes:
    - main_loop seeds opcode_ptr after compute_*_base so the
      first instruction (boot or resume from menu) has a
      correct pointer.
    - _ml_exit_code_cache calls update_opcode_ptr because
      opcode_ptr was pointing into CODE_CACHE_BUF before the
      transition and needs to be recomputed for bank 4 access.
    - _ml_normal_ptr drops its unconditional update_opcode_ptr
      + cs_in_attic-reentry block. ~70 cycles per non-attic
      instruction reclaimed — the biggest single emulator-side
      speedup available without restructuring the dispatch
      loop.

  Tested: DOS prompt, GWBASIC, EDIT.COM (fast and slow
  console), Multiplan, full FIXTEST suite (T1..T15), and
  DEBUG-T single-step through MOV/INC sequence — all green.

FIXTEST coverage
-----------------

Grew from 7 tests to 15 (T1..T15) plus a tiny child program
(T15CHLD.COM) for term_common coverage. Most tests added in this
sweep also exercise edge cases the bug originally hid behind
(e.g. T11 includes truncate-then-append on the same handle to
catch the post-truncate SFT-state bug; T12 includes both reject
and positive-path renames; T13 covers both AH=3C and AH=3D
handle scans).

Notable test-infrastructure lessons:
  - INT 21h handlers should preserve ES per the DOS contract;
    several didn't (notably AH=36h, AH=56h's pre-fix variant).
    Tests that use stosb after INT 21h calls now defensively
    push cs / pop es to stay robust against handlers that don't
    follow the contract.
  - .COM tests that EXEC a child must AH=4A-shrink first.

RECENT CHANGES (2026-04-20, part 3 — DEBUG deep polish)
=========================================================

DEBUG.COM transformed from a "capable first-pass clone" into a
near-complete DOS DEBUG replacement. Command set now covers the full
single-letter parity: A B C D E F G H I L M N O P Q R S T U W BP BL BC ?

Breakpoints (BP/BL/BC):
  - Up to 10 persistent software breakpoints via 0xCC patch/restore.
  - Shared infrastructure with P's step-over temp break.
  - Loop-back supported: BP at current target IP is skipped the first
    pass, then int1_handler single-steps through it and re-installs
    all BPs (via bp_install_all) so the target hits the same BP on a
    jump-back in the same G run.
  - Cleared on L / target reset.
  - stop_adjust_ip rolls IP back by 1 on planted-CC hit, leaves target
    IP alone on in-code INT 3.

Interactive A and E:
  - "A addr" / "E addr" with no payload enter DOS-DEBUG-style prompt
    loops; empty line exits, '-' backs up in E.
  - One-shot forms ("A addr <instr>", "E addr bytes...") still work.

R polish:
  - Register dump appends flag mnemonics (OV/NV DN/UP EI/DI NG/PL ZR/NZ
    AC/NA PE/PO CY/NC) after FL=XXXX.
  - "R F" enters flag edit; input is space-separated 2-char mnemonics.
  - Sub-register editing: R AL/AH/BL/BH/CL/CH/DL/DH with byte validation.

Loading:
  - "DEBUG FOO.COM arg1 arg2" auto-loads at startup via the PSP tail.
  - L, N, and auto-load all split filename from args identically and
    rebuild the target's PSP:80 tail + default FCBs (PSP:5C/6C) via
    the same rebuild_target_fcbs helper using DOS AH=29h.
  - MZ/ZM EXE loader: strips header, applies relocations, sets
    CS:IP/SS:SP from header, bounds-checks reloc targets into the
    loaded image, 32-bit file-size tracking (so large-on-disk EXEs
    with sub-64K code images work), header-size sanity checks.
  - All read paths detect short reads and reject instead of loading
    truncated or partially relocated images.
  - W [addr [len]]: explicit address/length for writing; short-write
    detection via AX vs requested length after AH=40h.
  - U [addr [end]]: second argument is now an inclusive end address
    matching DOS DEBUG (was being misread as a count).

Target process model:
  - init_target now allocates all available memory (AH=48 with BX=0xFFFF
    retry pattern) instead of a fixed 0x1000 paragraphs — EXEs with
    significant BSS or >64K requirements now fit.
  - Target PSP:02 (memory top) rewritten to match its own block size.
  - Target PSP:16 (parent PSP) points at DEBUG itself.
  - FLAGS initialized to 0x0202 (bit 1 is reserved-always-1).

Runtime integration with DOS:
  - run_target switches DOS's current PSP to target_seg via AH=50h
    before handing off; debug_stop_common restores DEBUG's PSP on stop.
    Target programs calling AH=51/62 now see their own PSP.
  - DTA is saved/restored per-target: run_target installs the target's
    remembered DTA (default PSP:0080, survives stops), stop path
    snapshots any AH=1A changes and restores DEBUG's DTA.
  - Terminate-family calls caught: INT 20h, AH=00h, AH=31h, AH=4Ch all
    stop DEBUG instead of bypassing to real DOS (where they would
    free/resize target_seg behind DEBUG's back).
  - target_terminated flag guards bare G/T/P after a terminate; user
    must supply an explicit address or L a fresh program to continue.
  - close_target_handles walks the target's JFT on a terminate-stop
    and closes handles 5-19 (normal DOS would do this on AH=4Ch —
    now DEBUG does it since we trap that call).

Emulator fix (required for DEBUG's trace):
  - 8086 TF timing corrected. Previously the opcode_done TF check
    used the *current* flag_tf, which self-triggered INT 1 on any
    IRET/POPF that loaded TF=1. Now flag_tf is snapshotted at
    instruction start into trap_flag_var, and opcode_done checks the
    snapshot — so T stops *after* the next instruction runs, matching
    real 8086. Fixes classic "DEBUG T leaves IP unchanged" behavior.

Other fixes along the way:
  - disasm_current was trashing its own return-value AX via crlf
    clobbering AL. Saved AX around crlf.
  - restore_debug_vectors was corrupting IVT[3/20/21] because the loop
    set DS to each old_int*_seg for the AH=25h install, then read the
    *next* vector's offset from that same DS. First INT 21 after
    restore (e.g. the "Execution stopped" print) then jumped through
    a trashed vector and rebooted. Reading via cs: overrides fixes it.

Known remaining gaps:
  - Narrow U disassembler (~30 opcodes recognized). Broader opcode
    coverage is the single biggest remaining DEBUG improvement.
  - Narrow A assembler parallel to U.
  - BP clear-by-address (only by index).
  - Files with both on-disk size AND load image >64K unsupported.

RECENT CHANGES (2026-04-20, part 2)
=====================================

GW-BASIC now runs on MegaDOS. Root cause: the startup probes INT 2Ah
(network / critical-section check) with AH=0 to detect a network
install. MegaDOS only installed INT 20h through INT 24h in the IVT,
so INT 2Ah pointed into the IVT table itself and GW-BASIC jumped to
garbage. Added a shared int_iret_stub and wired INT 28h (DOS idle)
and INT 2Ah to it. IRET preserves AH=0 so GW-BASIC sees "not
installed" and continues. (4e82889)

MS-DOS 3.3 batch vocabulary added to COMMAND.COM. Previously only
@/ECHO/REM/PAUSE worked in .BAT files. Now supported:

  - Labels:      ":name" lines are skipped (not echoed, not dispatched)
  - GOTO label:  scans batch_buffer for the label forward/backward,
                 prints "Label not found" and terminates on miss
  - IF [NOT] EXIST file cmd:        existence check via resolve_path
  - IF [NOT] ERRORLEVEL n cmd:      compares last_return_code >= n
  - IF [NOT] str1==str2 cmd:        literal compare, optional quotes
  - CALL cmd:    strips CALL prefix and re-dispatches (works for .COM;
                 nested .BAT would clobber batch state — known limit)
  - SHIFT:       rotates batch_args[1..9] → [0..8]

Per-line expansion runs after the @-strip, before dispatch:
  %0..%9   — args captured by batch_parse_args at run_batch entry
             (before resolve_path consumes the tail)
  %VAR%    — env_find_value, case-insensitive, unknown → empty
  %%       — literal %

Two plumbing bugs were caught while testing:
  1. env_find_value left ES=env_seg on return; batch_expand_line then
     stosb'd into the env block instead of expand_buf. Restore ES=CS
     after the call, re-fetch env_seg via [env_seg] for the copy.
  2. do_exec's .BAT fallback rewrote cmd_buffer in place with just
     "NAME.BAT\0", clobbering the args that lived in the same buffer.
     Now saves the tail to expand_buf first, then rebuilds name+tail
     before calling run_batch.

TESTBAT.BAT (uppercase name/content) and TESTBAT2.BAT (uppercase name,
lowercase content) are on the image and exercise every new feature
plus the case-insensitive command parser. (19e179f)

DEBUG.COM added — first-pass MS-DOS DEBUG clone. Command set:
C/D/E/F/H/I/M/O/S/R/A/U/G/T/P/?/Q. Hex-only, accepts OFFSET or
SEG:OFFSET, default segment tracks target program. Dump format is
16 bytes of hex per line (no ASCII column) to keep output narrow
enough for the MEGA65 terminal. (efb6724)

DEBUG P (proceed / step-over) partially implemented. For the
recognized step-over opcodes — CALL rel16, CALL ptr16:16, CALL
near/far via FF /2 /3, INT imm8, LOOPNZ/LOOPZ/LOOP, REP/REPNE
prefixes — setup_proceed_break plants INT 3 (0xCC) at IP+length,
clears TF, and runs free. When the break fires, the original byte
is restored and IP rolled back by one so it lands on the instruction
after the stepped-over call. Non-step-over opcodes fall through to
the TF path, which also works now (see next entry). (78610bd)

Emulator TF (trap flag) timing fixed. The TF check at opcode_done
previously read the *current* flag_tf, so an IRET/POPF that loaded
TF=1 self-triggered INT 1 before the next instruction ran — DEBUG's
T always stopped with IP unchanged. Now flag_tf is snapshotted at
the start of each instruction (into trap_flag_var) and the check at
opcode_done uses that snapshot. IRET that loads TF=1 doesn't self-
trigger; the first instruction of the traced code runs, and only
then does INT 1 fire. DEBUG T now single-steps correctly; non-step-
over P also works via the same mechanism.

DEBUG restore_debug_vectors DS corruption bug fixed. The loop set
DS to each old_int*_seg for AH=25h, then read the *next* vector's
offset from that same DS — fetching from whichever segment DOS was
last asked to install into. Subsequent AH=25h calls wrote garbage
to IVT[3]/[20]/[21]; the first INT 21h after restore (e.g. the
"Execution stopped" print) jumped through the trashed vector and
rebooted the emulator. Reading with cs: overrides keeps DS free to
carry the vector segment. (78610bd)

FORMAT.COM gained 160K/180K/320K support and floppy-label display
now disambiguates 5.25" DD formats by SPT+heads (1H/8SPT=160K,
1H/9SPT=180K, 2H/8SPT=320K, 2H/9SPT=360K). BIOS type byte stays
$01 for INT 13h AH=08 compatibility. (d6680ba)

RECENT CHANGES (2026-04-20, part 1)
=====================================

B: drive program execution now works reliably end-to-end.
Root cause turned out to be multiple bugs compounding each other:

1. load_drive_bpb was using dir_buffer as its scratch buffer for the
   boot-sector read. Cross-drive COPY stages file data in dir_buffer
   between read_cluster_data and write_cluster_data, so ensure_geo
   (triggered by the drive switch) silently overwrote the file data
   with the destination BPB, and we then wrote that BPB as the
   "file content." Fixed by giving load_drive_bpb its own bpb_buffer
   (512 bytes, separate from dir_buffer).

2. Floppy geometry state (floppy_a/b_spt/heads/cyls/bank/loaded) had
   been placed in a gap region of the PRG binary, so the .byte
   initializers were never emitted. floppy_a/b_bank stayed 0 at
   runtime, so every INT 13h DMA read from drive B went to attic
   $8000000 (guest RAM) instead of $8200000 (drive B image).
   Fixed by moving floppy state to $8F30-$8F3D and setting bank
   explicitly in load_floppy_drive.

3. Emulator floppy geometry at $8F26/$8F27 collided with the screen-
   clear row counters in do_clear_window/_scr. Any screen scroll
   silently clobbered floppy_b_spt/heads. Fixed by moving floppy
   state to $8F30-$8F3D.

4. Cross-drive COPY walked the source FAT chain using fat_buffer,
   but fat_buffer held the destination drive's FAT at that point.
   Fixed by swapping FAT buffers around fat12_next_cluster calls
   in the COPY loop.

5. detect_floppy_geom_drive defaulted to 1.44MB when the BPB was
   invalid (blank image). If the user mounted a 360K blank.img and
   ran FORMAT, FORMAT asked INT 13h AH=08 for geometry, got 1.44MB
   defaults, and formatted as 1.44MB — overflowing the 360K file.
   Fixed by pre-filling the attic slot with a $5A sentinel byte
   before loadfile_attic, then checking boundary bytes at each
   standard-size offset (160K/180K/320K/360K/720K) to determine
   the actual file size. Blank images of any size now format
   correctly, and mounted pre-formatted disks work as before.

6. Hardcoded cluster-allocation cap of 720 in six free-cluster
   search loops meant 720K/1.2MB/1.44MB media hit premature "disk
   full" during COPY/CREATE. Replaced with dynamic geo_max_clust
   so all supported disk sizes allocate their full capacity.

Additional fixes from the same session:
- INT 20h and INT 21h AH=4Ch now share common termination cleanup
  (term_common), so plain INT 20h / RET-to-PSP:0 exits also free
  memory and close handles.
- Vector save/restore re-enabled in int20_handler for shell-launched
  programs; previously skipped as a workaround.
- exec_drive saved separately from resolved_drive so PSP-setup
  INT 21h re-entry can't break the load path.
- Byte-safe pop of resolved_drive in read_cur_dir/write_cur_dir
  (was pushing/popping as word, overwriting batch_active).
- Drive-switch commands (A:/B:, CD cross-drive, AH=0Eh) now
  invalidate geo_active_drv so the next disk access re-reads
  the BPB of the new drive.
- _lfd_fail now clears the drive's loaded flag, so a failed
  remount doesn't leave sentinel garbage visible as "a disk."

Large refactor: consolidated every shared emulator address into
src/globals.asm (was src/zeropage.asm). Every $8Exx/$8Fxx scratch
variable now has a named constant in globals.asm, and every
source file references the name instead of a raw hex literal.
fat_writer.asm (built separately) also .includes globals.asm so
addresses never drift. This is what made the $8F26 collision
visible in the first place.

KNOWN ISSUES (FreeDOS)
=======================

FreeDOS boots successfully but is slow during startup due to
in-memory UPX decompression of the kernel. Once at the A:\>
prompt, performance is normal for the emulator's speed.

BIOS INTERRUPT STATUS
======================

INT 10h (Video) — BIOS (bios.asm, native mode):

FAST CONSOLE MODE
==================

Runtime toggle at $8F29: 1=fast (emulator intercepts INT 10h/21h/29h),
0=native (all INT 10h goes through BIOS IVT, refresh_cga copies vidbuf
to screen every tick).

Key difference: Fast mode writes to BOTH vidbuf ($18000) and screen RAM
($0800) directly. Native mode writes only to vidbuf via B800 mapping,
then refresh_cga copies to screen at ~18Hz.

The _i16_wait_key polling loop (io.asm) includes screen refresh and
cursor blink when fast console is OFF. Without this, the screen freezes
during keyboard waits because the 45GS02-level key poll blocks the
8086 decode loop (which is where the timer tick runs).

CLS works in both modes:
  - Fast mode: do_clear_window (vidbuf) + do_clear_window_scr (screen RAM)
  - Native mode: BIOS cls_vmem_clear_window fills B800 via REP STOSW

DISPLAY
========

Character set: CP437 8x8 font loaded from disk (cp437.bin) to
  character RAM at $02A000 (bank 2, ROM write-protect disabled).
  All 256 CP437 glyphs render correctly — box drawing, accented
  characters, block elements, etc.

Color RAM: MEGA65 color RAM at $1F800 used for reverse video.
  - Normal text: MONO_COLOR (configurable, default $05 = green)
  - Reverse video: MONO_REVERSE (MONO_COLOR | $20)
  - Color RAM scrolled/cleared with screen RAM in all paths

Video modes: TEXT 80x25 only (mode 3 CGA or mode 7 MDA).
  Set VIDEO_MODE in main.asm. No graphics modes.

Sprite cursor: 8x8 sprite tracks CGA cursor position.

refresh_cga (display.asm): Copies 2000 chars from vidbuf ($18000) to
  screen RAM ($0800), skipping attribute bytes (stride 2→1). Chars < $20
  are skipped (DMA fill pre-clears screen with $20/space). Runs at
  ~18Hz during timer tick AND during _i16_wait_key polling.

WORDSTAR STATUS
================

WordStar 3.x (WS.COM from WORDSTAR.IMG) partially works.

WORKING (fast console ON):
  - Opening menu displays correctly (text + function key bar)
  - CLS works
  - Screen clears between menus

BROKEN (fast console OFF / native mode):
  - Function key bar on row 24 shows garbled characters
  - Menu text area displays correctly (after many fixes)
  - CLS works

ROOT CAUSE ANALYSIS (row 24 garbled in native mode):
  WordStar writes DIFFERENT characters depending on mode. With fast
  console ON, it writes "1HELP..." (ASCII $31 $48 $45 $4C $50) with
  attributes $0F/$70. With fast console OFF, it writes $31 $EB $8D
  $EE $E8 with attributes $0F $E1 $1E $4A $F2.

  The garbled bytes ($EB E1 8D 1E EE 4A E8 F2) are an EXACT MATCH for
  WordStar's own code at CS:0F03. This means WordStar is choosing a
  different rendering path and using different character data.

  We confirmed:
  - The bar writer at CS:5040 always uses the SAME data table at CS:5067
    containing "1HELP 2INDENT..." in ASCII with interleaved attributes
  - But with fast console OFF, a DIFFERENT code path writes to row 24
    using AH=09 with char=$EB, attr=$E1 (confirmed via mem_write8 trap)
  - The AH=09 write comes from the BIOS's int10_write_char_attrib
    handler at F000:0BB5 (REP STOSW), called by WordStar's general
    character output routine at CS:2E52
  - WordStar's general writer at CS:2E52 is the BIOS fallback path,
    used when direct video access is not active
  - WordStar detects display type by reading BDA equipment word at 40:10,
    masking bits 4-5: $30=mono→B000, else→B800 (detection at CS:4FA0)
  - Detection code also reads config bytes at [02B3], [02D4], [02D5]
  - INSTALL.COM needed to configure terminal type but reports "too
    little memory" — may need more RAM or different DOS version

  UNSOLVED: Why does WordStar choose different output paths between
  fast console ON and OFF? Both modes return the same equipment word,
  video mode, and BDA values. The INT 10h handler behavior must differ
  in some subtle way that WordStar detects. Possible causes:
  - Timing: BDA repair runs at different times relative to WS startup
  - AH=02 cursor positioning behaves differently
  - AH=09 cursor advance behavior differs (fast mode doesn't advance,
    BIOS mode doesn't advance — should be the same)
  - Some other INT 10h function returns different values
  - WordStar may do a direct B800/B000 write-then-read test that
    fails differently in one mode

  NEXT STEPS for debugging:
  - Try running INSTALL.COM with more RAM or different DOS version
  - Add trap on WordStar's detection code at CS:4FA0 to see what
    values it reads from BDA in each mode
  - Check if the "general writer" at CS:2E52 is ONLY used in native
    mode (suggesting WS detected no direct video access)
  - Check [44CE] (stored video segment) and [44D1] (direct video flag)
    after WS startup in both modes

KEYBOARD MAPPING
=================

Arrow keys:    MEGA65 PETSCII → IBM PC scancodes (up=$48, down=$50,
               left=$4B, right=$4D) with AL=0 (extended key)
Backspace:     PETSCII $14 / ASCII $7F → AL=$08, AH=$0E
Enter:         $0D → AL=$0D, AH=$1C
TAB:           Reserved for emulator menu (not passed to guest)
Other keys:    Passed through as ASCII, scancode=$00

SOUND
======

PC speaker emulated via SID chip. PIT Timer 2 frequency divider
converted to SID frequency: (1193472 / divider) << 4. Port $61
bit 0+1 gate speaker on/off. SID sustain/release for clean tones.

TIMING
=======

Real-time timer tick via MEGA65 frame counter ($D7FA).
Fires every 3 frames (~16.7 Hz at 50Hz, ~20 Hz at 60Hz).
BDA tick counter at 40:6C incremented at ~18 Hz.
INT 8 delivered via int8_asap flag checked at opcode_done.

DISK I/O
=========

Dual floppy support (drive A and B). Images loaded to attic RAM:
  Drive A: $8100000, Drive B: $8200000
Auto-detect floppy geometry from BPB (360K/720K/1.2M/1.44M).
Dirty tracking per drive. FAT32 file writer for saving to SD card
  (implemented but disabled — jmp _sfd_end in disk.asm).

MEGADOS PATH SEARCH
====================

PATH command sets the executable search path. When a command is not
found in the current directory, the shell searches each PATH directory
for .COM, .EXE, and .BAT files. Case-insensitive PATH= env lookup.
Child programs inherit command line arguments via PATH search.

Example: PATH A:\BIN;A:\UTILS

MEGADOS I/O REDIRECTION
========================

Implemented: >, >>, < for built-in commands and external programs.
Shell output goes through INT 21h AH=02h which checks if stdout
is a device or file. AH=09h also routes through AH=02h.
Child programs inherit parent JFT (including redirected handles).
Refcounts incremented for file handles (5+) during JFT inherit.

Limitations:
  - No pipe (|) support
  - No stderr (2>) redirection
  - No quoted filenames or filenames with spaces
  - Programs using INT 10h directly bypass redirection
  - Redirected output is slow (one byte at a time through AH=40h)
  - AH=06h output still intercepted by emulator (bypasses redirection)

ARCHITECTURAL LIMITATIONS
==========================

- Device handle I/O is not fully generic. When AH=02h (and other
  character output calls) detects stdout is a device, it always
  routes through the CON driver — not through whatever driver is
  bound to that specific handle. This means if stdout were redirected
  to AUX or PRN via handle manipulation, output would still go to CON.
  The proper fix requires adding a driver pointer to each SFT entry
  and restructuring how handles map to device drivers. Handles 0-4
  are always bound to specific devices (stdin/stdout/stderr=CON,
  stdaux=AUX, stdprn=PRN), so this only matters for non-standard
  handle reassignment — extremely rare on a floppy system.

KNOWN BUGS
===========

- AH=41h (delete file) fails when called from a child program
  (returns error 2 / file not found). Works correctly from shell
  built-in DEL command. Root cause unknown — likely related to
  resolve_path or cmd_buffer interaction with child segment context.
  Workaround: use AH=13h (FCB delete) from programs.

- Emulator data cache can corrupt tail end of loaded program memory.
  Workaround: place critical strings earlier in program data section.

- Running DOSTEST twice may show minor handle leaks (refcount drift
  for device handles inherited across program invocations).

- Stray keypress after menu resume: occasionally after returning from
  the TAB menu to the emulation, a phantom keypress is prepended to
  the next command, causing "Bad command or file name". Multiple drain
  attempts at both the emulator ($D610) and DOS (AH=0Ch) levels have
  not fully resolved it. Suspected MEGA65 keyboard hardware timing —
  key events may arrive after the drain loop completes. Affects any
  running program, not just the shell.

- Ctrl-C during nested EXEC: if program A launches program B via
  AH=4Bh, Ctrl-C in B jumps straight to cmd_loop, skipping A's
  cleanup and return. Should return to A instead.

- Ctrl-C during file I/O: when BREAK=ON, MS-DOS checks for Ctrl-C
  on all INT 21h calls including AH=3Fh/40h (file read/write).
  We only check during console I/O (AH=01h/02h/0Ah).

- BREAK=ON adds con_check_input overhead to every AH=02h output
  call. May slow console output noticeably. Could throttle to
  check every Nth character if performance is an issue.
